## Sign Off Ready <a name="Sign-Off-Ready"></a>

A component in Figma is considered "signed off" and ready to publish to other designers when it has been reviewed and approved by the relevant stakeholders, usually including designers, developers, and product managers, and is considered to be in a stable state with consistent functionality and visual appearance, with any necessary documentation readily available.

Key indicators that a component is ready to publish:

<table>
  <tr>
    <td>Step</td>
    <td>Description</td>
  </tr>
  <tr>
    <td>Design Review Process</td>
    <td>
      The component has gone through a formal design review process where feedback is
      gathered and addressed
    </td>
  </tr>
  <tr>
    <td>Documentation</td>
    <td>
      Detailed documentation is created explaining the component&#39;s usage,
      accessibility considerations, and any specific design guidelines
    </td>
  </tr>
  <tr>
    <td>Version Control</td>
    <td>
      The component is marked as &quot;ready for dev&quot; or a similar status within your
      design system, indicating it&#39;s finalized and ready for implementation
    </td>
  </tr>
  <tr>
    <td>Team Alignment</td>
    <td>
      There is clear communication across the design team that the component is ready for
      use in live projects
    </td>
  </tr>
</table>

## How to manage component approval in Figma <a name="Manage-Component-Approval"></a>

<table>
  <tr>
    <td>Feature</td>
    <td>Description</td>
  </tr>
  <tr>
    <td>Design System Library</td>
    <td>
      Organize components within a dedicated Figma library and publish only
      &quot;ready&quot; components
    </td>
  </tr>
  <tr>
    <td>Component Status Tags</td>
    <td>
      Use labels or tags to indicate the current state of components (e.g., &quot;In
      Progress&quot;, &quot;Review&quot;, &quot;Approved&quot;).
    </td>
  </tr>
  <tr>
    <td>Branching Feature</td>
    <td>
      Utilize Figma&#39;s branching feature to create separate development branches for
      components under refinement, publishing only from the &quot;master&quot; branch.
    </td>
  </tr>
  <tr>
    <td>Feedback Mechanisms</td>
    <td>
      Leverage Figma&#39;s commenting and annotation features to provide feedback during
      the review process.
    </td>
  </tr>
</table>

## Changes to designed and live pages <a name="Changes-To-Designed-And-Live-Pages"></a>

- Once we ship or have major changes, all changes should be branched, even adding a button or text
- Snowflakes stay in CDS Project called BetaNXT Snowflakes
- If a UI pattern is used more than three times, turn it into a reusable UI component.
- If a UI component is used in 3 or more projects/teams, put it in your design system.

## Buttons <a name="Buttons"> </a>

- Different buttons have different use cases.
- Border radius 4px

### Primary Buttons <a name="Primary-Buttons"></a>

- Primary button is designed for a user to perform the most important task on the page, the “Done” button to show that you have completed the task in this case

### Secondary Buttons <a name="Secondary-Buttons"></a>

- Secondary buttons are usually the alternative to the primary action. Use secondary buttons for less important actions or for multiple actions that have equal importance.
- It is paired with a primary button. The secondary button serves as the negative action of the pair, such as “Delete,” "Cancel" or “Remove”.
